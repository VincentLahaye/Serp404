# Serp404 — Design Document

## Overview

Serp404 is an open-source (MIT) desktop application that helps SEO professionals and webmasters find broken pages (404s) among URLs indexed by Google for any given domain. It collects URLs from multiple sources, verifies their Google indexation status, audits their HTTP health, and exports results as CSV.

## Target Audience

- SEO consultants auditing client websites
- Webmasters monitoring their own sites
- Anyone needing to find 404 errors in Google's index for a domain

## Core Workflow (4 Phases)

### Phase 1 — URL Collection

Collect candidate URLs from up to 3 sources:

1. **serper.dev** (`site:domain.com` queries with pagination) — URLs returned are **confirmed indexed** by Google
2. **Sitemap.xml** — Auto-fetch and parse `sitemap.xml` (including nested sitemaps). Indexation status: **unknown**
3. **CSV Upload** — Smart import that auto-detects URL columns from Screaming Frog, Ahrefs, SEMrush, or any CSV. Indexation status: **unknown**

All URLs are deduplicated and merged into a single list per project.

### Phase 2 — Indexation Verification

- URLs discovered via serper.dev `site:` search are already marked as **confirmed indexed**
- URLs from sitemap/CSV that were NOT found in serper results need individual verification
- Verification: query serper.dev with `site:domain.com/exact-path` for each unconfirmed URL
- User controls this phase: can skip it (save API credits) or run it
- Shows estimated credit cost before starting

### Phase 3 — SEO Audit

For each confirmed indexed URL, check:

- **HTTP status code** (200, 301, 302, 404, 500, etc.)
- **Response time** (in ms)
- **Redirect chain** (follow all hops, detect loops)
- **Page title** (from `<title>` tag, detect empty titles)

Features:
- Adjustable concurrency (1-50 concurrent requests) via UI slider, changeable in real-time
- Start/Pause/Stop controls
- Real-time progress with live-updating results table
- Resumable: progress is saved to SQLite after each URL, survives app restart

### Phase 4 — Results & Export

- Summary dashboard: total URLs, % by status code, count of 404s, redirects, empty titles, slow pages
- Filterable/sortable results table with quick filters: All | 404 | Redirections | Empty Titles | Slow (>2s)
- CSV export with all collected data

## Tech Stack

### Desktop Framework: Tauri v2

- Cross-platform: Windows, macOS, Linux
- Lightweight: ~10-15 MB installed (uses OS native WebView, no bundled Chromium)
- Native event system for real-time frontend ↔ backend communication
- Auto-update support

### Frontend: React + Vite + Tailwind CSS

- Dark mode modern UI (Linear/Raycast style)
- Background: `#0a0a0f`, cards with subtle borders, green/blue accent color
- React with TypeScript
- Tailwind CSS for styling
- Vite for fast dev builds

### Backend: Rust (via Tauri)

- `reqwest` + `tokio` for async HTTP requests with adjustable concurrency (`Semaphore`)
- `rusqlite` for embedded SQLite database
- `quick-xml` or `roxmltree` for sitemap XML parsing
- `csv` crate for CSV import/export
- `serde` for serialization

### Database: SQLite (embedded)

Zero config, single file, survives app restart.

## Data Model

```sql
CREATE TABLE projects (
    id TEXT PRIMARY KEY,
    domain TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT 'created',
    created_at TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE urls (
    id TEXT PRIMARY KEY,
    project_id TEXT NOT NULL REFERENCES projects(id),
    url TEXT NOT NULL,
    source TEXT NOT NULL,              -- 'serper' | 'sitemap' | 'csv'
    indexed_status TEXT DEFAULT 'unknown', -- 'confirmed' | 'not_indexed' | 'unknown'
    http_status INTEGER,
    response_time_ms INTEGER,
    title TEXT,
    redirect_chain TEXT,               -- JSON array of redirect hops
    error TEXT,                        -- Error message if request failed
    checked_at TEXT,
    UNIQUE(project_id, url)
);

CREATE TABLE settings (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
```

## UI Screens

### Screen 1 — Home (Project List)

- Header: "Serp404" logo + Settings gear icon
- "+ New Project" button
- Grid of existing projects showing: domain, creation date, status, mini-stats

### Screen 2 — New Project (Modal)

- Domain input field
- Source checkboxes:
  - Sitemap.xml (auto-detected)
  - serper.dev (disabled if no API key configured)
  - CSV Upload (drag & drop zone)
- "Start Collection" button

### Screen 3 — Project View (3 tabs)

**Tab "Collection"**: Progress per source, real-time URL count, scrollable discovery log

**Tab "Indexation"**: List of unconfirmed URLs, "Verify via serper" button with cost estimate, live progress

**Tab "Audit"**: Concurrency slider, Start/Pause/Stop, global progress bar, real-time stats, live results table, quick filters, CSV export button

### Screen 4 — Settings

- serper.dev API key field (masked, with "test key" button)
- Default preferences: timeout, user-agent, default thread count

## BYOK Model

All API keys are provided by the user (Bring Your Own Key):
- Stored locally in SQLite `settings` table
- Never transmitted anywhere except to the respective API (serper.dev)
- Displayed masked in the UI with copy/test functionality

## Key Technical Decisions

1. **Tauri over Electron**: ~10x lighter, Rust backend is ideal for concurrent HTTP work
2. **SQLite over Postgres**: Zero external dependency, perfect for single-user desktop app
3. **Tauri events over WebSockets**: Native, zero config, bidirectional communication built-in
4. **tokio::Semaphore for concurrency**: Allows real-time adjustment of concurrent request count
5. **Smart CSV import**: Scan all columns, detect URL patterns (regex), let user confirm which column to use
